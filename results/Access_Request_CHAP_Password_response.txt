**Threat Model:**

- **Attack Assumption:**  
  - RADIUS is deployed over UDP/1812 as in RFC 2865, without transport protection (no IPsec, no RadSec/TLS/DTLS).  
  - PPP CHAP is used, so NAS sends `Access-Request` packets containing `CHAP-Password` (Type 3) to the RADIUS server.  
  - The attacker can passively monitor the network path between NAS and RADIUS server (e.g., same L2 segment, SPAN port, compromised router).  
  - The attacker does *not* know any RADIUS shared secrets and cannot force the server to accept forged `Access-Accept` packets.

- **Attacker Capability:**  
  - CAN:
    - Sniff and record complete RADIUS `Access-Request` and `Access-Accept`/`Access-Reject` packets.  
    - Extract arbitrary attributes from the captured packets (`CHAP-Password`, `CHAP-Challenge`, `Request Authenticator`, `User-Name`, etc.).  
    - Perform unlimited offline computation (dictionary/brute‑force) on captured data.  
  - CANNOT:
    - Modify packets in flight without detection by higher-layer behavior (i.e., we assume no active tampering; attack is purely passive plus offline computation).  
    - Learn RADIUS shared secrets from configuration.  
    - Directly authenticate to the RADIUS server as a NAS client if not configured as one.

---

**Attack Procedure:**

1. **Capture a CHAP-based Access-Request.**  
   - The attacker passively monitors RADIUS traffic and waits for an `Access-Request` that uses CHAP: it will contain `CHAP-Password` (Type 3) and *not* `User-Password` (per RFC 2865 §4.1).  
   - The attacker records:
     - `User-Name` (username).
     - `CHAP-Password` (CHAP Identifier + 16‑octet response).
     - `CHAP-Challenge`, if present, *or* the `Request Authenticator` field if `CHAP-Challenge` is absent (per RFC 2865 §2.2, §5.3, §5.40).

2. **Parse CHAP-Password and the challenge from the packet.**  
   - From RFC 2865 §5.3, the `CHAP-Password` attribute format is:
     - 1 octet CHAP Ident (call this `ID`).
     - 16 octets CHAP Response (call this `RESP`).
   - The attacker extracts:
     - `ID` = first byte of the `CHAP-Password` attribute’s value.
     - `RESP` = next 16 bytes of that value.
   - The attacker determines the challenge `CHAL` as:
     - If `CHAP-Challenge` attribute (Type 60) is present: use its value.  
     - Otherwise: `CHAL` = 16‑octet `Request Authenticator` from the RADIUS header (RFC 2865 §2.2, §5.3, §5.40).

3. **Recreate the CHAP verification function offline.**  
   - RFC 2865 §2.2 describes the server’s CHAP verification:  
     - It “encrypts the challenge using MD5 on the CHAP ID octet, that password, and the CHAP challenge ... and compares that result to the CHAP-Password.”  
   - In PPP CHAP terms, the response is:  
     - `RESP = MD5( ID || password || CHAL )`  
       where `password` is the user’s *cleartext* password stored by the RADIUS server (§2.2 notes the password must be available in cleartext).

4. **Run an offline dictionary / brute-force attack on the user’s password.**  
   - Using the captured `(ID, CHAL, RESP)` triple and the known function:
     - For each candidate password `pw_guess` in a dictionary or keyspace, compute:  
       `TEST = MD5( ID || pw_guess || CHAL )`
     - If `TEST == RESP`, then `pw_guess` is the correct cleartext password with overwhelming probability.
   - This attack:
     - Does *not* require the RADIUS shared secret at any point.  
     - Is entirely offline; rate-limiting, lockout, or monitoring on the RADIUS server cannot detect it.  
     - Can be accelerated using standard MD5 cracking hardware (GPUs, FPGAs).

5. **Use the recovered password to impersonate the victim.**  
   - Once the attacker recovers `password` for a given `User-Name`:
     - The attacker can initiate their own PPP CHAP session (through any NAS under their control or a compromised one) using `User-Name` and `password`, and the RADIUS server will accept if policy allows.  
     - If the same credentials are reused elsewhere (VPN, Wi‑Fi, AD/LDAP, web SSO, etc.), the attacker can leverage them for broader compromise beyond RADIUS.

6. **Why this attack specifically leverages CHAP-Password:**
   - The `CHAP-Password` attribute carries the raw CHAP MD5 response *without* being encrypted or mixed with the RADIUS shared secret (unlike `User-Password`, which is XORed with MD5(secret||RequestAuthenticator), RFC 2865 §3).  
   - The challenge (`CHAP-Challenge` or `Request Authenticator`) is transmitted in cleartext as well.  
   - Therefore, a passive eavesdropper has *all* inputs to the CHAP verification function except the password and can directly mount an offline guessing attack.  
   - By contrast, when PAP is used via `User-Password`, a passive attacker without the RADIUS shared secret cannot trivially recover the cleartext password from the captured attribute. With CHAP-Password, the RADIUS-layer secret provides no additional protection against offline guessing; security is reduced to the strength of the password alone.

**Attack Message:**

```json
{
  "Code": 1,                          // Access-Request
  "Identifier": 0x5a,
  "Length": 80,
  "Authenticator": "0x3f4a9c1b72e4d0a6b58c9e23d4f0a1b2",
  "Attributes": [
    {
      "Type": "User-Name",
      "Value": "alice"
    },
    {
      "Type": "CHAP-Password",        // Attribute 3
      "Value": {
        "CHAP-Ident": "0x07",         // 1 octet
        "CHAP-Response": "0x11223344556677889900aabbccddeeff" // 16 octets
      }
    },
    {
      "Type": "CHAP-Challenge",       // Attribute 60 (optional, else challenge is Authenticator)
      "Value": "0xa1b2c3d4e5f60718293a4b5c6d7e8f90"
    }
  ]
}
```

- From this captured packet, the attacker derives:
  - `ID = 0x07`
  - `RESP = 0x11223344556677889900aabbccddeeff`
  - `CHAL = 0xa1b2c3d4e5f60718293a4b5c6d7e8f90`
- Then brute-forces to find `password` such that:  
  `MD5( 0x07 || password || CHAL ) = RESP`.
